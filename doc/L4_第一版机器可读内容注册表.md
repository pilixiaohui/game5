# L4 第一版机器可读内容注册表

| 项目 | 内容 |
| --- | --- |
| 文档层级 | L4，第一版内容部署与静态契约 |
| 权威主题 | `content.v1.0`内容集合的机器入口、字段所有权、加载顺序、作用域隔离和静态失败条件 |
| 当前状态 | 106项内容身份与引用已经登记；注册表静态加载通过；玩法行为、组合生成、参数平衡和发布证据仍未完成 |
| 适用范围 | Godot第一版配置加载、无表现模拟、页面查询、保存版本和内容扩充 |
| 直接依赖 | `L4_内容语法与登记规则.md`、四份第一版L4内容文档、七份L3契约、`L6_平衡参数与首小时规模验证.md` |
| 机器入口 | `res://data/content/v1/content_manifest.json` |
| 加载与校验 | `res://scripts/content/V1ContentRegistry.gd` |
| 静态测试 | `res://tests/l6/l4_content_contract_tests.gd` |
| 不负责 | 正式参数值、玩法实例状态、场景表现资源、实际组合可完成性和玩家体验结论 |

本文不重新叙述各房间、单位、节点或候选的设计理由。四份第一版L4内容文档继续负责语义、取舍和验收场景；本注册表负责把其中可执行的身份、引用、作用域和接口关系变成唯一机器输入。

## 1. 权威边界

第一版内容事实按以下边界维护：

| 事实 | 唯一所有者 |
| --- | --- |
| 允许发生的状态变化、原子边界与守恒 | 对应L3契约 |
| 内容语义、设计理由、组合约束与验收场景 | 对应第一版L4内容文档 |
| 内容ID、类别、作用域、所有者、交叉引用、参数引用和运行资格 | 本机器注册表 |
| 参数类型、量纲、夹具、候选范围、发布值和证据 | `data/balance/v1_parameter_registry.csv.txt` |
| 页面布局、文本组织、动画和画面代表 | L5与表现资源 |
| 当前库存、任务、单位、节点、候选组和轮次 | 权威存档状态 |

同一个结构化字段不得在场景、`.tres`或脚本中维护第二份运行值。人读文档中的清单是语义说明和审计入口；一旦其ID、作用域或引用与注册表不一致，构建必须失败并要求同一变更同时修订二者，不能选择其中一份静默继续运行。

## 2. 注册表包

注册表使用一个清单和四个领域分片：

| 文件 | 内容 | 定义数 |
| --- | --- | ---: |
| `data/content/v1/content_manifest.json` | 版本、分片、允许类别、作用域、L3所有者、预期数量和禁止输入 | 清单 |
| `data/contracts/v1/development_contract_manifest.json` | 表达式、接口、查询、T1/T2验收和冻结基线入口 | 开发契约清单 |
| `data/content/v1/hive_content.json` | 核心巢室、六个可建设房间、飞升入口和配方图 | 22 |
| `data/content/v1/unit_content.json` | 通用效果、己方/敌方/中立单位和过程资产 | 30 |
| `data/content/v1/world_content.json` | 生态、空间段、节点职责和世界来源 | 22 |
| `data/content/v1/evolution_content.json` | 候选来源、里程碑、科技族、候选、永久收益和领袖 | 32 |

清单固定预期总数为106。分片丢失、重复、额外定义、类别数量变化或内容版本不一致都属于配置失败。增加或删除内容必须明确提升内容版本或按兼容变更流程更新预期数量，不能依赖目录扫描自动纳入未知文件。

## 4. 通用外壳

每个定义显式保存：

```json
{
  "content_id": "v1.room.biomass_filter",
  "kind": "room",
  "content_version": "content.v1.0",
  "display_name_key": "content.room.biomass_filter.name",
  "display_name_zh": "腐殖滤囊",
  "scope": "required",
  "design_status": "已确认机制",
  "evidence_status": "mapped_not_implemented",
  "owner_contract": {
    "contract_id": "HRC",
    "interface_version": "L3-1.2"
  },
  "tags": [],
  "reads": [],
  "writes_via": [],
  "parameter_refs": [],
  "unlock_refs": [],
  "presentation_class": "hive_room_resource",
  "acceptance_refs": [],
  "runtime_enabled": true,
  "spec": {}
}
```

数组没有内容时写空数组；单值不存在时写`none`。加载器不补显示名、所有者、参数、行为、配方边、死亡输出或作用域默认值。

`spec`按类别保存最低字段。类别要求由`V1ContentRegistry.REQUIRED_SPEC_FIELDS`执行，主要包括：

- 房间的兼容键、槽位、配方、端口、功率策略、可观察状态和保留事实。
- 配方的输入、消费边界、功率、时长、输出、容量承诺、取消边界、谱系和后继边。
- 单位的阵营、移动形态、合法位置、耐久、唯一行为、目标组、效果、变形、死亡输出和批次键。
- 生态、空间段、节点职责和来源的合法组合、基础完成内容、服务边界与恢复规则。
- 里程碑、科技族、候选、永久收益和领袖的作用接口、来源、代价与只读边界。

所有计算值使用`runtime-expression.v1.0`结构化AST。`*_ref`只能保存单一稳定参数ID；绑定值和组合值必须写入`*_expr`，不得在字符串中保存运算符、等级变量或模板占位符。

## 5. 作用域隔离

| `scope` | `runtime_enabled` | 允许行为 |
| --- | --- | --- |
| `required` | `true` | 可以进入第一版生成、命令和模拟 |
| `optional` | `true` | 可以进入合法支路，但不能成为基础恢复或通关唯一钥匙 |
| `preview_only` | `false` | 只允许查询和显示，`writes_via`必须为空 |

注册表加载成功不代表所有`required`内容已经实现。它只表示内容关系可解析。运行目录编译器还必须检查目标构建支持该内容的L3接口；缺少实现时阻止发布配置，不能删除定义后继续运行。

内容清单必须同时解析`runtime-expression.v1.0`、`l3-interface.v1.0`和`query-dto.v1.0`。全部`reads`、`writes_via`和`*_expr`均通过开发契约交叉校验；只有内容字段齐全但接口或表达式未知时仍属于配置失败。

## 6. 加载顺序

启动配置按以下原子顺序建立：

```text
读取content_manifest.json
-> 校验清单版本、分片路径、L3所有者和参数表路径
-> 读取四个分片
-> 合并并检查106项全局唯一ID
-> 校验公共字段、类别最低字段和作用域
-> 校验全部内容引用与类别
-> 校验配方有向图、单位行为和谱系
-> 读取活动参数表并校验全部parameter_refs
-> 形成只读运行目录
-> 才允许创建新轮次或载入存档
```

任一步失败时进入配置错误页。禁止启动半份内容、跳过错误分片、用脚本常量补值或自动把预览内容改为运行内容。

## 7. 参数状态

当前活动参数表包含342个参数身份。新增的内容参数全部满足：

- `status = fixture_only`。
- `release_value`为空。
- 没有候选范围时`candidate_min/max`为空。
- `evidence_version = none`。
- `fixture_value`只保证类型可解析和机制测试可建立，不代表难度、节奏或发布平衡。

因此当前注册表可以用于静态引用校验和后续无表现夹具，不能用于宣称首小时曲线、百万级节奏、领袖差异或飞升预览已经平衡。发布配置仍必须拒绝空发布值。

## 8. 静态校验覆盖

当前加载器直接检查：

- 清单、分片、版本、预期数量和全局ID唯一性。
- ID类别一致、小写规范、本地化键和中文名存在。
- L3所有者存在且接口版本为`L3-1.2`。
- 所有内容引用存在，关键引用的目标类别正确。
- 所有参数引用在活动参数表中存在且参数ID不重复。
- 配方图无递归环，配方可暂停、具有取消边界和明确输出。
- 单位只有零或一个核心行为，行为引用合法效果，效果属于六类通用效果。
- 过程资产具有谱系；只读内容不能写状态或进入运行池。
- 生态不声明按当前构筑、军队或最近失败自适应的输入。
- 注册表不把动画、画面代表、帧率、视口或页面访问作为玩法输入。

以下内容仍必须由生成器、无表现内核或真实页面测试证明，不能由静态JSON冒充：

- 任意合法首轮组合均可完成且保留能源、工蜂、样本和出口恢复闭环。
- 基础单位在真实空间、出兵和损耗规则下能够完成每类必需节点。
- 候选修改在运行内核中不增加第二行为或资源捷径。
- 生成器和运行脚本确实不读取当前构筑、军队、最近失败或表现状态。
- 动画、代表数量和帧率变化不会改变实际结算。

## 9. 查询接口

`V1ContentRegistry`只提供只读用途：

| 接口 | 作用 |
| --- | --- |
| `load_registry()` | 加载并完整校验注册表包 |
| `has_definition(id, kind)` | 检查稳定身份和类别 |
| `get_definition(id)` | 返回深复制，不能修改注册表原件 |
| `ids_for_kind(kind, runtime_only)` | 按稳定ID排序查询类别 |
| `runtime_definitions()` | 返回排除只读预览后的定义深复制 |
| `registry_summary()` | 返回版本、数量、来源文件和错误摘要 |

领域服务不得根据任意字符串动态调用方法。`writes_via`是允许接口声明，实际命令和效果仍通过显式分派表进入L3。

## 10. 存档与版本

存档保存`content_version = content.v1.0`和实例引用的稳定`content_id`，不复制完整定义。载入必须满足：

1. 存档内容版本可由当前构建直接读取或具有显式迁移器。
2. 每个实例ID存在且类别与保存位置兼容。
3. 所需参数版本和规则版本匹配。
4. 预览内容没有运行实例。
5. 缺失内容不会由显示名、数组位置或相似ID猜测替代。

内容更名只改本地化键或显示文本。改变类别、核心行为、物质谱系、原子边界或作用域时建立新ID或提升内容版本，并提供存档迁移。

## 11. 扩充流程

新增内容必须在同一个变更中完成：

1. 在对应L4文档说明设计关系、取舍和验收场景。
2. 在对应分片加入完整定义并更新清单预期数量。
3. 在活动参数表登记全部参数身份，正式值继续走L6证据流程。
4. 增加或更新内容静态测试和领域无表现测试。
5. 验证基础完成与恢复不依赖新增可选内容。
6. 验证存档版本兼容或提供迁移器。
7. 取得对应证据后再把内容加入发布运行池。

只增加场景节点、脚本常量或`.tres`资源而没有注册内容身份，属于未登记内容，构建必须失败。
