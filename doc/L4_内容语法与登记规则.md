# L4 内容语法与登记规则

| 项目 | 内容 |
| --- | --- |
| 文档层级 | L4，内容语法 |
| 权威主题 | 内容定义身份、字段、组合、参数引用、作用接口与第一版内容完整性 |
| 当前状态 | 内容语法为`已确认规则`；第一版内容关系为`已确认机制`；具体数值为`待平衡参数` |
| 适用范围 | 第一版及后续横向内容扩充 |
| 直接依赖 | `L1_第一版范围与完成定义.md`、`L2_形式系统总模型.md`、`L2_统一术语表.md`、七份L3契约 |
| 被引用 | 其余L4内容文档、两份L5文档、L6参数与测试 |
| 对应决定 | D-009、D-021、D-025、D-035至D-050 |
| 未取得证据 | 内容引用完整性、首轮生成覆盖、基础内容可完成性、候选差异与玩家可读性 |
| 替代来源 | `06`、`07`、`08`、`09`、`23`、`26`、`29`中的内容定义；旧文件继续保留设计理由，不再作为内容注册源 |

本文只回答“哪些具体内容使用已经确认的机制，以及这些内容必须登记什么”。它不重新定义资源、房间、单位、地图、候选、保存或飞升状态机，也不规定Godot中的节点、场景和脚本结构。

## 1. 权威边界

第一版内容由四份语义文档和一份机器注册表共同组成，双方的字段所有权不同：

| 文档 | 权威内容 |
| --- | --- |
| `L4_第一版虫巢房间与配方.md` | 固定基础设施、六个可建设房间、配方图、端口和内容解锁 |
| `L4_第一版单位与战斗内容.md` | 幼体、己方单位、敌方单位、可破坏结构、残骸和通用效果配置 |
| `L4_第一版世界与关卡内容.md` | 生态、节点职责、空间段、首轮组合和相邻区域概要 |
| `L4_第一版候选解锁与飞升预览内容.md` | 科技族、候选、里程碑、普通来源、永久收益和领袖预览目录 |
| `L4_第一版机器可读内容注册表.md` | 机器入口、字段所有权、加载顺序、数量和失败条件 |
| `data/content/v1/content_manifest.json`及四个分片 | 内容ID、类别、作用域、所有者、交叉引用、参数引用和运行资格的唯一机器值 |

发生冲突时使用以下顺序：

1. L0和L1决定产品目标与第一版体验边界。
2. L2决定对象、所有权、状态和全局不变量。
3. L3决定系统允许的状态变化和跨域接口。
4. 本文决定内容可以怎样组合L3接口。
5. 其余L4文档决定第一版内容语义、关系、取舍和验收场景。
6. `content.v1.0`机器注册表决定可执行结构化字段；与人读L4不一致时阻止构建，不能静默选择一方。
7. L5只决定如何组织触达、页面、反馈和表现，不能修改内容效果。
8. L6决定参数值是否可晋升以及内容是否通过验证。

旧编号文档中的同名对象若与L4冲突，以L4为准。L4没有登记的对象不能因为场景中存在同名节点、旧资源文件或占位图标而进入第一版玩法。

## 2. 内容、参数、实例与表现

四类事实必须分离：

| 层 | 示例 | 是否由L4拥有 |
| --- | --- | --- |
| 内容定义 | `采质工蜂`能使用一个搬运资产行为 | 是 |
| 参数 | 工蜂携带量、周期、最大耐久 | 只登记参数ID，值由L6拥有 |
| 权威实例 | 某批工蜂当前数量、位置、耐久和携带资产 | 否，由L2/L3状态拥有 |
| 表现投影 | 场景里显示多少只工蜂、播放哪段动画 | 否，由L5表现映射拥有 |

内容定义不得保存当前库存、任务进度、节点控制、候选组成员或本轮突变实例。参数表不得增加新行为、新状态或新资源来源。表现资源不得成为产出、命中、死亡或解锁条件。

## 3. 稳定身份与版本

### 3.1 ID格式

第一版使用以下稳定形式：

```text
v1.<内容域>.<英文短名>
```

允许的首批内容域：

| 内容域 | 对象 |
| --- | --- |
| `ecology` | 区域生态原型 |
| `room` | 固定基础设施或可建设房间蓝图 |
| `recipe` | 房间任务、持续网络贡献或恢复配方 |
| `unit` | 己方、敌方与中立单位模板 |
| `asset` | 过程资产与残骸定义 |
| `effect` | 通用伤害、状态、变形、出兵或修正配置 |
| `segment` | 长战场空间段原型 |
| `node_role` | 节点职责锚点与组合约束 |
| `source` | 外部物质、环境输入或特殊战利品来源 |
| `milestone` | 行为里程碑定义 |
| `tech_family` | 科技族资格 |
| `candidate` | 候选定义 |
| `permanent` | 永久收益预览定义 |
| `leader` | 虫群领袖预览定义 |

ID只表达稳定身份，不编码等级、当前数值、随机权重或本地化名称。内容更名不改变ID；改变对象类别、核心行为或守恒关系必须建立新ID并迁移旧引用。

### 3.2 内容版本

第一版内容集合版本记为`content.v1.0`。每个定义至少保存：

- `content_id`。
- `content_version`。
- `display_name_key`与第一版中文名。
- `scope`：`required / optional / preview_only / excluded`。
- `design_status`：只使用宪章规定的`已确认机制 / 待平衡参数 / 第一版排除 / 长期设想 / 已废弃`。
- `evidence_status`：由L6登记实现、模拟、试玩和发布证据，不能反向改变设计状态。
- `owner_contract`与接口版本。
- `parameter_refs`。
- `presentation_class`，只作L5映射引用。
- `runtime_enabled`，并与`scope`严格一致。
- 对应类别的完整`spec`。

存档快照保存内容集合版本和实例引用。载入时找不到定义、引用到错误对象类别或缺少必需参数，属于配置或迁移失败，不能用默认内容补齐。

## 4. 通用定义外壳

所有内容定义使用同一外壳：

| 字段 | 规则 |
| --- | --- |
| `content_id` | 全局唯一且稳定 |
| `kind` | 本文登记的内容域之一 |
| `content_version` | 第一版固定为`content.v1.0` |
| `display_name_key / display_name_zh` | 稳定本地化键和第一版中文回退名 |
| `scope` | 明确第一版必需、可选、只读预览或排除 |
| `tags` | 只使用术语表和L3允许的规范标签 |
| `owner_contract` | 唯一写入实例状态的L3契约 |
| `reads` | 只读条件与来源接口 |
| `writes_via` | 允许调用的L3命令、请求或效果类型 |
| `parameter_refs` | 全部可调量的唯一参数ID |
| `unlock_refs` | 发现、资格和可执行性引用，不使用单一“已解锁”布尔值 |
| `presentation_class` | L5可引用的表现类别，不包含结算逻辑 |
| `acceptance_refs` | L6内容、接口和体验验收ID |
| `runtime_enabled` | 运行内容为`true`；只读预览为`false`且不得声明写接口 |
| `spec` | 对应内容类别的最低字段，不允许脚本补默认值 |

单值为空时必须显式写`none`，集合为空时写空数组，不能依赖脚本默认值。只有表现类别可以在不改变玩法内容版本的前提下替换资源。

## 5. 房间与配方语法

### 5.1 房间蓝图最低字段

每个房间定义必须在L3房间最低字段之外登记：

- 恰好一个主`room_family`、`structure_compatibility`和`group_compatibility_key`；能源、资源、解析、孵化、回收、储存、支援等附加职责使用规范`tags`和配方表达，不建立多个互相竞争的主分类。
- 允许的槽位标签、等级、分支和运行绑定。
- 一个或多个`recipe_ref`，以及每个配方的循环资格。
- 物理、网络、单位、候选端口及本地缓冲类型。
- 结构与运行功率请求的安全层级和服务模式。
- 缺料、缺能、堵塞、待机和运行的内容特有可观察来源。
- 建设、移动、合并、拆分、重组、替换、降级和拆除时需保留的绑定事实。
- L5使用的建设、运行、阻断和合并表现类别。

房间定义不能增加专用任务生命周期。差异必须由配方图、端口、容量、环境条件、绑定或通用修正形成。

### 5.2 配方最低字段

| 字段 | 含义 |
| --- | --- |
| `recipe_id` | 稳定配方身份 |
| `task_kind` | 结构任务、过程任务或持续网络贡献 |
| `inputs` | 核心资源预留、过程资产锁、单位输入或环境服务 |
| `consumption_boundary` | 预留、开始、渐进或完成时消费 |
| `power_request` | 服务模式、安全层级和参数引用 |
| `duration_ref` | 单一时间参数ID；持续网络贡献写`continuous`；使用`duration_expr`时写`none` |
| `outputs` | 物理、网络、单位或候选输出之一或多个 |
| `capacity_commitments` | 输入、输出、单位、端口和候选槽承诺 |
| `cancel_boundary` | 取消可以释放和不能返还的事实 |
| `lineage_rule` | 外部、内部或恢复来源谱系 |
| `next_recipe_refs` | 后续配方边；没有则写`none` |

输入、功率和输出的单一参数值使用`amount_ref`；需要绑定或计算时使用`amount_expr`。`duration_expr`、`amount_expr`以及永久收益的`current_value_expr`、`next_value_expr`、`cost_expr`和`stack_boundary_expr`必须符合`T1_运行目录表达式契约.md`，禁止保存公式字符串或隐式模板占位符。

配方图必须是可验证的有向图。循环只由房间运行策略重新建立任务，不能在图中形成无法暂停或无法保存的递归事件。

## 6. 单位、行为与效果语法

### 6.1 单位模板最低字段

每个单位模板必须完整登记L3要求的：阵营、标签、移动形态、合法位置、空间占用、耐久、伤害与抗性标签、零或一个核心行为、被动范围、死亡输出、控制资格、无目标规则和批次合并键。

此外必须登记：

- `behavior_ref`或`none`。
- 有序`target_groups`，不得由页面临时注入。
- 允许产生的六类通用效果引用。
- 变形来源/目标及继承白名单。
- 第一版参数ID集合。
- 独立目标条、批次代表或背景流量的表现身份。

一个候选可以修改已有行为条件、目标、周期或效果，但不能给同一模板增加第二个核心行为。需要新动词时必须建立新模板或新形态。

### 6.2 行为定义

行为只组合以下公共步骤：

```text
过滤目标 -> 稳定选择 -> 接近 -> 校验空间/输入/容量
-> 累积进度 -> 形成一种通用效果草案 -> 按L3阶段提交
```

第一版允许的效果类别固定为：`damage / carry_asset / transform_self / spawn_unit / apply_status / create_process_asset`。移动、待命、死亡、撤离响应和被动范围不是额外核心行为。

## 7. 世界、节点与来源语法

### 7.1 生态原型

生态原型登记：

- 兼容节点职责池、空间段池、敌方池、来源池和环境池。
- 必需、可选和互斥标签。
- 世界概要可公开的主题、资源倾向和压力维度。
- 节点网与长战场的参数引用。
- 禁止组合及安全骨架引用。

生态原型不能读取当前军队、构筑或最近失败来改变内容答案。

### 7.2 节点职责

节点职责只声明组合要求，不建立另一种关卡状态机。每个职责至少登记：

- `node_role_id`、必需标签和可选附加标签。
- 允许的空间段、敌人、阻断目标、来源和特殊对象。
- 最低可见目标与基础单位可达要求。
- 占领后开放的来源、环境或修正接口。
- 情报层级允许显示的字段。
- 是否可跳过、是否是出口、是否参与恢复闭环。

Boss、资源、样本、精英、虫巢和出口都是标签组合，不建立专用节点底层。

### 7.3 来源定义

外部物质来源必须登记储量或流量、抽取容量、形成位置、过程资产谱系和恢复规则。环境输入必须登记环境容量、兼容房间和服务条件。特殊战利品必须登记具体死亡输出或处理链，不能写“节点完成奖励”。

## 8. 候选、解锁与长期目录语法

### 8.1 行为里程碑

里程碑只读取已经提交的事件。定义至少包含：

- 稳定里程碑ID与一次性/可重复边界。
- 允许的事件类型、过滤标签和来源对象。
- 更新的发现、科技族资格或内容池资格。
- 最早生效步骤。
- 是否永久化；第一版默认只在本轮有效。

页面访问、提示关闭、动画播放和现实等待不能满足里程碑。

### 8.2 候选

候选除L3最低字段外必须登记：

- 作用接口和被修改的内容字段。
- 明确收益关系与新增压力，不使用“全面增强”。
- 允许来源、来源权重参数和同化成本模式。
- 重复、叠加、互斥与目标仍合法时的失败边界。
- L5中必须显示的作用对象、收益、代价和来源事实。

### 8.3 永久收益与领袖

第一版只登记可复算的预览目录。每项必须有目标接口、当前值表达式、购买后值表达式、成本表达式、叠加边界表达式和已登记显示规则。全部表达式由运行目录编译一次并同时供领域与页面查询使用。领袖必须登记强化、弱项、独有机制、专属候选池和全部参数ID；领袖不拥有地图生成修正字段。

缺少发布参数时预览显示配置未完成并阻止发布，不允许显示静态示例冒充真实值。

## 9. 参数引用规则

- 所有成本、产量、容量、时长、概率、权重、耐久、伤害、射程、倍率和表现配额都必须使用唯一参数ID。
- L4只能说明参数的单位、作用边界和必须验证的问题，不能把机制夹具值写成正式内容值。
- 参数可以为空，但空发布值必须被L6配置校验识别为发布阻断。
- 一个参数改变对象身份、行为数量、状态枚举、来源谱系或原子边界时，它不是参数，应回到L2/L3/L4变更流程。
- 内容不得在Godot资源、场景或脚本中添加登记表之外的隐藏玩法常量。

## 10. 横向扩充规则

新增内容按以下顺序接入：

1. 选择现有内容域和L3所有者。
2. 复用现有端口、行为、效果、节点标签或候选作用接口。
3. 说明它与现有内容形成的不同关系，而不只是更大数值。
4. 登记全部参数ID和表现类别。
5. 检查基础恢复路径不依赖该新内容。
6. 加入内容引用、生成合法性和无表现对照测试。
7. 取得证据后才进入发布内容池。

以下情况必须先提升机制版本，不能作为普通内容扩充：

- 新核心资源或新过程资产阶段。
- 第二个单位核心行为。
- 新命令阶段或专用战斗结算。
- 绕过房间形成资源的节点奖励。
- 允许多主动战场。
- 修改基础撤离、保存或飞升原子边界。

## 11. 内容完整性校验

L6至少执行以下静态校验：

| ID | 校验 |
| --- | --- |
| L4C-001 | 所有内容ID唯一，引用目标存在且对象类别正确 |
| L4C-002 | 所有第一版必需内容都具有L3所有者和完整最低字段 |
| L4C-003 | 所有可调量只通过已登记参数ID引用 |
| L4C-004 | 所有房间配方图可达、可暂停、可取消并有明确输出端 |
| L4C-005 | 所有单位至多一个核心行为，效果只属于六类通用效果 |
| L4C-006 | 所有死亡输出、外部来源和特殊战利品具有物质谱系 |
| L4C-007 | 所有必需节点可以用基础必需单位完成 |
| L4C-008 | 首轮任意合法内容组合保留能源、工蜂、首次样本和出口恢复闭环 |
| L4C-009 | 所有候选目标在第一版存在，收益和压力作用于合法接口 |
| L4C-010 | 只读预览和排除内容不能进入命令、账目或完成依赖 |
| L4C-011 | 内容随机池不读取当前构筑、军队或最近失败生成答案 |
| L4C-012 | 表现资源、动画和实体配额不被任何玩法条件引用 |

## 12. L4完成条件

L4只有同时满足以下条件才算达到第一版设计完整：

- 第一版范围中的每个房间、单位、敌人、节点、来源、候选和预览对象都具有稳定定义。
- 每个定义都能追溯到一个L3接口，不存在内容专用隐式状态机。
- 每个可调量都有参数身份，即使发布值仍待测试。
- 任意首轮合法组合都能由生成校验器证明可达、可完成和可恢复。
- L5可以只引用内容ID和权威事实完成页面与进程设计，不需要从旧编号文档猜测规则。
- L4C-001至L4C-012必须映射到L6；在证据通过前，内容关系可以是`已确认机制`，但不能称为已验证内容，具体数值仍为`待平衡参数`。
