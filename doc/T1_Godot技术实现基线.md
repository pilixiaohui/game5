## 1. 工具链基线

| 项目 | 第一版基线 |
| --- | --- |
| 引擎 | Godot `4.7.stable`官方版本 |
| 脚本 | GDScript 2.0；第一版运行不依赖外部语言运行时 |
| 玩法数值 | 显式整数或浮点参数；时间权威按L3统一步骤，不读取动画帧 |
| 内容格式 | 版本化JSON注册表；参数使用活动CSV；场景和表现资源不得保存第二份玩法值 |
| 存档格式 | L3规范化JSON信封、版本字段和校验和；磁盘写入使用临时文件原子替换 |
| 自动化 | Godot无窗口脚本测试；测试命令必须能在未打开编辑器的环境执行 |
| 发布平台 | 以`T2_桌面运行基线.md`定义的Windows桌面构建为第一门槛 |

项目升级Godot小版本前必须运行内容注册、L3内核、确定性、存档往返和桌面冒烟测试。涉及序列化、随机、线程、容器迭代或渲染后端变化时不能只验证编辑器能打开项目。

## 2. 权威运行原则

### 2.1 唯一状态根

一次活动游戏会话只能拥有一个权威状态根。第一版以`L3HeadlessEngine.state`所代表的形式状态为目标实现，不允许同时维护：

- UI自己的资源库存。
- 场景节点自己的单位真实数量。
- 房间动画自己的任务完成状态。
- 地图按钮自己的占领布尔值。
- 另一个自动加载单例中的候选或战场副本。

页面、动画、粒子、画面代表和调试面板都是查询投影。它们可以缓存展示模型和插值进度，但不能成为保存来源或结算依据。

### 2.2 单向写入

```text
输入设备 / 页面意图
-> 应用命令适配器
-> L3命令信封
-> 权威模拟步骤与领域服务
-> 原子状态提交 + 事件日志
-> 查询快照
-> 页面与表现投影
```

任何页面不得持有领域服务的可变字典引用。任何表现回调不得直接调用资源增加、单位死亡、节点占领或候选生成函数。

### 2.3 确定性输入

权威结果只读取：

- 当前规范状态。
- 已验证内容目录和参数版本。
- 已提交命令及稳定接收顺序。
- 对应领域的确定性随机流。
- 推进的统一模拟步骤数。

帧率、窗口尺寸、页面停留、动画完成、音频回调、实体代表数量、系统语言和容器偶然迭代顺序不得进入结果。

## 3. 分层与依赖方向

第一版采用六层结构：

| 层 | 职责 | 可以依赖 | 禁止依赖 |
| --- | --- | --- | --- |
| Bootstrap | 组合对象、校验配置、选择新游戏/载入路径 | 下列所有层的公开构造接口 | 玩法细节和页面节点路径 |
| Content | 加载注册表、参数和本地化，编译只读运行目录 | 文件适配器、L4/L6数据 | 当前会话状态和页面 |
| Domain | L3状态、命令、步骤、事件、随机和不变量 | 只读运行目录、标准容器 | Node、场景、输入、动画和磁盘 |
| Application | 会话生命周期、命令提交、查询、自动保存和离线推进协调 | Domain、Content、Persistence接口 | 具体页面控件 |
| Persistence/Platform | 文件、时钟、线程、日志和操作系统适配 | Godot平台API | 领域规则 |
| Presentation | 场景、页面、视图模型、声音和动画 | Application查询与事件 | Domain可变状态和磁盘实现 |

依赖只能向内。Domain层必须能在`--headless`下实例化，不创建场景树，不访问`Input`、`DisplayServer`、`AudioServer`、`Time`或`user://`。

## 4. 目录基线

现有文件可以分阶段迁移，不要求为了改名一次性移动全部代码；新代码按以下职责放置：

```text
data/
  content/v1/                 # content.v1.0注册表包
  balance/                    # 活动参数与隔离的旧探针参数
  validation/                 # 机器验收映射
scripts/
  bootstrap/                  # AppRuntime与启动配置
  content/                    # 注册表加载、参数加载、运行目录编译
  simulation/l3/              # 纯L3权威内核
  application/                # SessionController、命令/查询门面
  persistence/                # SaveRepository、迁移器、设置仓储
  presentation/               # 页面控制器、视图模型和投影器
  platform/                   # 桌面时钟、线程、窗口和日志适配器
scenes/
  bootstrap/                  # 最小启动场景
  shell/                      # 全局游戏外壳与弹层宿主
  pages/                      # 虫巢、地图、战场、进化、虫群
  presentation/               # 房间、单位代表、特效和音频场景
tests/
  l3/                         # 领域契约和不变量
  l4/                         # 内容静态与组合测试
  integration/                # 会话、保存和页面命令集成
  desktop/                    # 分辨率、输入和截图验收
```

`scripts/simulation/v1/`、当前硬编码目录和当前主场景只作为迁移探针，不得被新页面当作第一版权威接口。

## 5. 启动与组合根

### 5.1 启动顺序

```text
读取最小应用设置
-> 建立日志与崩溃上下文
-> 加载并校验content.v1.0
-> 加载活动参数身份和所选参数配置
-> 编译运行目录
-> 建立领域内核与会话控制器
-> 载入存档索引
-> 显示标题页
-> 新游戏或载入成功后创建游戏外壳
```

内容或参数配置失败时只能进入配置错误页和日志导出入口，不能创建半初始化会话。

### 5.2 自动加载边界

目标工程只保留一个应用组合根`AppRuntime`作为全局入口。它拥有但不公开可变引用：

- `ContentRegistry`。
- `RuntimeCatalog`。
- `SessionController`。
- `SaveRepository`。
- `SettingsRepository`。
- 日志、音频和窗口适配器。

页面只调用`AppRuntime.commands`、`AppRuntime.queries`和会话生命周期请求。当前`ConfigDB`、`GameState`、`SimulationService`和旧`SaveService`自动加载在迁移完成后移除；迁移期间不得让新旧服务同时写同一会话。

音频、窗口和设置服务可以作为组合根子节点存在，但不能持有或改写领域状态。

## 6. 内容与参数装配

### 6.1 唯一机器入口

内容从以下入口加载：

```text
res://data/content/v1/content_manifest.json
```

加载器必须先完成`L4_第一版机器可读内容注册表.md`定义的全部静态校验，再编译运行目录。运行目录只包含`runtime_enabled = true`定义；只读预览仍可由查询目录读取，但不能被命令分派器解析为可执行内容。

### 6.2 参数配置

`res://data/balance/v1_parameter_registry.csv.txt`是第一版参数身份唯一入口。运行配置分为：

| 配置 | 读取列 | 用途 |
| --- | --- | --- |
| `fixture` | `fixture_value` | 守恒、接口和组合测试 |
| `experiment` | 预注册覆盖集 | 批量模拟与形成性试玩 |
| `release` | `release_value` | 发布候选和正式构建 |

发布配置遇到任一`fixture_only`、空`release_value`、证据版本不匹配或超出适用构建范围时必须失败。脚本中的`1.0`只能用于无语义数学常量或明确测试输入，不能补内容成本、周期、容量、伤害、概率、倍率或表现配额。

### 6.3 运行目录编译

原始JSON不直接在高频模拟中反复查找。`RuntimeCatalogCompiler`目标职责：

1. 把稳定字符串ID映射为会话内只读索引。
2. 解析并类型检查参数引用。
3. 预编译目标组、标签位集、配方边和合法效果分派。
4. 分开运行内容和只读预览内容。
5. 计算内容、参数和规则组合哈希。
6. 返回不可由页面修改的目录视图。

数值编译严格执行`T1_运行目录表达式契约.md`和`runtime-expression.v1.0`。编译器输入仅为已校验内容、明确参数配置、表达式契约和白名单上下文；输出必须包含类型、单位、参数依赖、上下文依赖、绑定槽和稳定节点序列。禁止解析公式字符串、执行动态表达式或在领域与页面各实现一份公式。

GDScript字典没有真正只读语义，因此目录内部不外露；查询接口返回值对象或深复制。存档继续保存稳定字符串ID，不保存会话内索引。

## 7. 命令、查询与事件

### 7.1 命令

页面产生意图，不构造领域结果。每个命令通过L3命令信封保存：

- `command_id`。
- `command_type`。
- `issuer`。
- `targets`。
- `parameters`。
- 接收边界与稳定序号。
- 最早执行步骤。
- 幂等键。
- 生命周期。

应用层维护显式命令分派表。未知命令、目标类别错误、预览内容、过期拓扑版本或非法参数立即返回结构化拒绝，不使用反射调用任意方法。

命令信封中的`command_type`使用`l3-interface.v1.0`稳定ID；运行方法名只存在于`interface_registry.json`的`runtime_type`及显式适配映射。请求、任务、效果、事件、事实读取和拒绝码同样必须先登记。

### 7.2 查询

查询只读取已提交状态并返回页面所需的最小快照，例如：

- 全局资源与能源摘要。
- 虫巢拓扑、房间群、任务、端口和阻断原因。
- 世界、区域、节点情报和合法命令能力。
- 战场真实数量、空间、战线、损耗、残骸和来源。
- 候选组、资格、成本、突变和飞升预览。

查询可以按页面形成专用视图模型，但字段必须追溯到权威事实。页面不得通过多个查询结果自行推导占领、支付、伤亡或完成结论。

第一版页面只能调用`query-dto.v1.0`登记的16个查询。请求字段、返回DTO、空值、错误、分页、状态版本和缓存失效规则以`T1_查询DTO契约.md`为准；不得返回领域可变字典。

### 7.3 事件

事件日志用于：

- 里程碑与跨域请求。
- 页面增量刷新和表现触发。
- 自动保存脏标记。
- 调试回放和验收证据。

事件是已提交事实，不是第二条命令通道。表现消费事件后不能回调领域服务重复形成结果。页面错过事件时必须能从最新查询快照重建正确状态。

事件公共载荷、71种事件类型及查询失效组以`l3-interface.v1.0`为准。`before/after/details`只用于诊断，页面不得把它们当作稳定DTO。

## 8. 模拟循环

### 8.1 统一步骤

权威模拟只调用L3的十二阶段顺序。Godot的`_process`、`_physics_process`和Tween只负责请求推进与显示插值，不能替代模拟阶段。

推荐应用循环：

```text
采集本帧输入意图
-> 提交命令信封
-> 根据单调时间计算应推进的完整步骤数
-> 在本帧工作预算内推进确定性步骤
-> 提交查询快照和事件批次
-> 页面插值到最新快照
```

若待推进步骤超过交互帧预算，显示“同步中”并分块推进；不得丢弃步骤或通过加大产量近似补偿。

### 8.2 在线、失焦和离线

窗口失焦不改变玩法规则。应用记录最后提交步骤和时间证据；进程被暂停或关闭后，重新进入时通过L3离线边界计算可推进步骤。系统时钟只用于计算候选经过时间，实际结果仍由逐步或已证明等价的批量算法产生。

### 8.3 时间倍率

如果第一版开放时间倍率，它只能改变单位现实时间内请求推进多少统一步骤，不改变步骤内容、任务成本、随机顺序或动画完成条件。页面暂停不能作为全局模拟暂停；玩家仍只暂停房间和循环等设计允许的对象。

## 9. 百万级状态实现

### 9.1 禁止逐单位节点

真实单位数量与画面实体数量分离：

- 领域层使用数量批次、耐久队列、位置、空间和状态键保存真实数量。
- 表现层在预算内生成少量代表、背景流量和固定簇。
- 一个代表节点不拥有真实耐久、目标、死亡输出或资源。
- 代表被回收、遮挡或离屏不能触发领域死亡。

百万级状态不得创建百万个`Node`、`Resource`或每步对象。高频容器优先使用稳定数组、紧凑结构、数量段和可证明无损的商余算法。

### 9.2 批次稳定性

合并键必须来自注册表中的模板、阵营、位置、空间、耐久、行为、状态和来源事实。批次合并、拆分、保存载入、表现配额或线程分片不得改变：

- 目标槽分配。
- 同时伤害结果。
- 出生和死亡总量。
- 根网按真实锚点数量形成的功率。
- 资产谱系和死亡输出。

具体算法与预算继续以`L6_百万级状态队列与性能预算.md`为权威。

## 10. 线程与帧响应

第一版遵循“单一逻辑提交者”：

- 权威状态只由一个顺序执行上下文提交。
- 页面和Godot对象只在主线程访问。
- 常规步骤先在主线程按工作预算执行，避免无证据的并发复杂度。
- 离线长推进、批量验证或纯查询预计算可以进入`WorkerThreadPool`，但输入必须是深复制或不可变数据，任务不得访问场景树。
- 工作线程只返回结果草案；最终版本检查和原子提交仍回到单一逻辑提交者。

只有性能证据证明主线程预算无法达到桌面门槛时，才把常规模拟迁入专用线程。迁移必须补充命令队列、快照交换、取消、关闭和确定性对照测试。

## 11. 场景与页面边界

### 11.1 场景树职责

目标场景结构：

```text
Bootstrap
  AppShell
    GlobalStatusBar
    PrimaryNavigation
    PageHost
    ActiveBattleStrip
    ModalLayer
    NotificationLayer
    DebugOverlay (debug only)
```

页面通过`PageHost`按L5导航切换。科技候选、房间建设、房间群详情、撤离确认和飞升预览使用专用页面或弹层，不把全部操作堆入同一页。

### 11.2 页面控制器

每个页面控制器只负责：

1. 订阅所需查询版本。
2. 把查询映射到控件和表现投影。
3. 收集用户意图并提交命令。
4. 显示命令接受、拒绝和待提交状态。
5. 在离开页面时释放表现资源。

控制器不得计算产量、伤亡、候选抽取、节点完成或飞升点。房间生产动画只读取任务阶段、进度、服务功率、缓冲和工蜂交互事件。

### 11.3 投影稳定性

页面切换、缩放、滚动、对象池回收、动画强度和减少闪烁设置只能重建投影。视图模型必须以稳定ID为键更新，不能依赖场景子节点数组位置代表领域顺序。

## 12. 保存、载入与迁移

### 12.1 保存信封

保存信封至少包含：

- 格式与快照版本。
- 构建版本。
- `rules_version`。
- `content_version`。
- 参数、生成和飞升公式版本。
- 权威状态负载。
- 随机流状态。
- 命令接收序号和幂等记录边界。
- 规范负载校验和。
- 最后提交步骤与离线时间证据。

`L3SnapshotCodec`的规范化与校验和规则是当前实现起点。任何压缩或外层元数据都不能改变规范负载哈希。

### 12.2 原子写入

```text
生成并校验内存快照
-> 写入同目录临时文件
-> 刷新并关闭
-> 重新读取临时文件并校验
-> 保留上一份有效备份
-> 原子替换目标槽
```

保存失败不覆盖上一份有效存档。自动保存和手动保存共用同一仓储与格式，不维护两套序列化。

### 12.3 载入与迁移

载入顺序为：信封结构、校验和、版本支持、内容引用、参数身份、不变量、迁移、再验不变量。迁移器按版本逐级执行，必须幂等且保留原文件备份。未来版本存档、未知内容ID、错误类别或无迁移路径必须明确拒绝。

## 13. 错误、日志与调试

### 13.1 错误类别

| 类别 | 玩家结果 | 开发信息 |
| --- | --- | --- |
| 配置错误 | 阻止进入游戏，允许查看错误摘要 | 文件、字段、ID、版本和校验阶段 |
| 命令拒绝 | 保持状态，页面显示事实原因 | 命令ID、目标、拒绝码和步骤 |
| 不变量失败 | 停止当前会话提交并尝试保留诊断快照 | 前后哈希、最近命令与事件 |
| 保存失败 | 保留旧存档并提示重试 | 路径、系统错误和校验结果 |
| 表现失败 | 降级或重建投影，不改玩法状态 | 场景、资源和稳定ID |

面向玩家的文本不显示堆栈和内部公式；日志必须足以让开发者复现。发布日志不得记录操作系统账户名或不必要的个人路径。

### 13.2 调试能力

调试构建至少提供：

- 当前步骤、规则/内容/参数版本和状态哈希。
- 命令与事件过滤查看。
- 资源、单位和过程资产守恒审计。
- 查询指定稳定ID的定义与实例。
- 固定种子重建与命令回放。
- 表现代表数量与真实数量对照。

调试入口不得存在于发布导航，也不得通过修改状态绕过正式命令后被当作验收证据。

## 14. 本地化与设置

- 运行逻辑只保存稳定ID，不保存中文显示名作为键。
- 注册表`display_name_key`进入Godot翻译表；`display_name_zh`用于首版回退和静态审计。
- 缺失翻译显示明确缺键标记并在发布检查中失败，不能回退到内容ID后继续验收。
- 音频、窗口、文本大小、减少闪烁和动画强度保存在独立设置文件，不进入玩法快照哈希。
- 设置改变表现后必须能从当前查询快照重建，不要求重新载入存档。

## 17. 当前迁移边界

当前工程必须明确区分：

| 已可复用 | 仍需替换或接入 |

## 18. 技术完成定义

Godot第一版技术实现只有同时满足以下条件才算完成。机器状态登记在`res://data/validation/t1_technical_acceptance_manifest.csv.txt`；下列十项对应`T1D-001`至`T1D-010`，三份配套契约还必须通过`T1C-001`至`T1C-014`：

1. `[T1D-001]`新进程和载入存档都先通过内容、参数、版本和不变量校验。
2. `[T1D-002]`所有玩法状态只有一个权威根，页面和场景不保存第二份结算事实。
3. `[T1D-003]`所有页面写操作都能追溯到L3命令；所有结果能追溯到事件和状态差异。
4. `[T1D-004]`106项内容由注册表驱动，发布运行不存在硬编码第二目录或隐藏玩法常量。
5. `[T1D-005]`百万级单位使用无损批量状态，不按真实数量创建Godot节点。
6. `[T1D-006]`保存、读档、离线、回放和不同帧率保持确定性。
7. `[T1D-007]`预览内容不能进入命令、账目、轮次重置或完成依赖。
8. `[T1D-008]`当前旧全局服务不再与正式会话并行写状态。
9. `[T1D-009]` `T2_桌面运行基线.md`全部发布门槛通过。
10. `[T1D-010]`L1、L4、L5和L6要求的对应发布证据真实存在，不能用此技术清单替代。
