## 1. 六类接口

| 类别 | 方向 | 是否进入命令队列 | 是否可直接改变状态 |
| --- | --- | --- | --- |
| `command.*` | 页面或明确系统控制器 -> 应用层 -> 领域 | 是 | 只有提交阶段可以 |
| `request.*` | 一个领域所有者 -> 另一个领域所有者 | 否 | 由接收方形成草案或结果 |
| `task.*` | 已提交命令/配方 -> 持续过程 | 否 | 只按任务生命周期和统一步骤 |
| `effect.*` | 行为或任务 -> 同步效果草案 | 否 | 只在登记提交阶段 |
| `event.*` | 领域 -> 日志、应用和表现 | 否 | 否，事件是已提交事实 |
| 事实读取接口 | 内容资格与领域规则 -> 权威状态 | 否 | 否 |

内容定义中的`reads`和`writes_via`必须逐项存在于本注册表。前缀存在但完整ID未登记仍视为未知接口。

## 3. 命令结果

统一结果结构：

```text
command_id
status = received | pending_validation | accepted | rejected | committed | cancelled
rejection_code = none | 已登记拒绝码
committed_event_ids[]
result_snapshot_version
```

- `rejected`不得产生预留、消费、随机游标推进或部分事件。
- `committed_event_ids`只能引用同一命令实际形成的已提交事件。
- 长时任务创建成功后命令为`committed`；任务后来堵塞不是把原命令改成失败。
- 注册表之外的拒绝字符串是接口错误，必须先登记并提升契约版本后使用。

缺字段、未知字段、类型错误、非法发起者、目标类别错误、过期版本和预览内容写入使用机器登记的七个结构拒绝码。它们发生在领域命令入队之前，与资源不足、资格不足等领域拒绝分开。

## 4. 请求、任务和效果

请求不是可以从页面伪造的第二套命令。每个请求只有登记所有者可以发起，接收方只返回草案、承诺或回执。任务只能由已提交命令、合法配方或恢复保障创建；效果必须经过统一阶段的资格、草案和提交边界。

第一版注册12项请求、6项任务和13项效果。新增内容可以复用这些接口横向扩充；如果内容需要新的状态写入种类，必须先新增L3接口和机器登记，不能在`spec`中放入任意方法名。

## 5. 事件契约

事件类型统一保存为不带前缀的运行类型，例如机器接口`event.unit_died`在事件信封中写`unit_died`。映射规则固定为移除`event.`前缀，不允许别名。

事件公共字段为：

```text
event_id, event_type, step, phase, global_sequence,
source_id, target_id, parameter_version, one_shot_key
```

`before`、`after`和`details`只用于诊断与验收，不是页面DTO。表现收到事件后只按其`invalidation_group`使相关查询失效，再读取最新查询快照。页面不得依靠某个可选`details`字段计算伤亡、支付、占领或任务完成；错过事件也必须能从查询恢复。

机器注册表完整登记当前71种事件及其所有者和失效组。领域代码发出未登记事件、内容引用未登记事件或同一运行类型映射到多个接口时均为配置错误。

## 6. 事实读取接口

内容`reads`使用132个精确事实读取ID。它们只表示领域规则可以读取哪类权威事实，不是页面查询，也不暴露可变字典。运行目录编译时把每个ID绑定到显式只读访问器；未知ID阻止配置。

页面只能调用`T1_查询DTO契约.md`登记的查询，不能直接调用这些细粒度事实访问器。

## 7. 版本与扩充

以下变化必须提升`l3-interface`次版本并提供迁移或兼容决定：

- 新增或删除接口ID、命令字段、事件类型或拒绝码。
- 改变字段类型、必填性、提交阶段、所有者或运行映射。
- 改变事件失效组或让诊断字段成为公共字段。

只增加显示文本、测试证据或不改变接口的实现优化不提升接口版本。
