## 1. 必守语义

1. 真实单位数量是非负整数，不按画面实体数截断。
2. 每个逻辑个体仍具有精确耐久、状态和行为语义，但同态个体由数量聚合表示。
3. 只有完整玩法键相同的单位可以合并。
4. 拆分、合并、保存、离线、批次布局、容器顺序和线程不能改变结果。
5. 不允许平均耐久、取平均持续时间、裁剪低频状态、随机抽样伤亡或把尾部队列计入“其他”。
6. 性能预算可以拒绝发布内容配置，也可以让未发布工作跨现实帧计算；不能改写已经合法存在的玩家状态。
7. 无法在安全看门限内完成时，放弃本次未发布步骤并进入诊断暂停，保留上一稳定边界，不得提交近似结果。

## 2. 精确数量与数值表示

- 单位、命中槽、状态层数和来源份额使用经过溢出检查的64位非负整数语义。
- 任一事务将超过`2^63-1`时完整拒绝并报告表示范围错误，不饱和、不回绕、不转浮点。
- 耐久、伤害、行为进度、持续时间和倍率求值使用参数版本声明的整数或定点量子。
- 发布参数必须能够无损转换到权威量子；浮点只可用于表现插值和非权威预览。
- 取整位置由L3公式声明，同一运算不能因批次大小改变取整次数。

百万级只增加整数数量的位宽，不应线性增加运行对象数。

## 3. 规范状态单元

### 3.1 定义

`规范状态单元`是状态队列中的最小聚合项，保存`完整玩法键 -> 真实数量`。完整玩法键只包含会改变后续规则的事实：

- 模板、阵营、移动形态和内容/参数版本。
- 唯一真实位置、节点/空间段、空间层与活动。
- 精确剩余耐久量子。
- 规范状态效果多重集与当前派生有效修正键。
- 核心行为定义、精确行为进度、目标池/输出位置、路径和最早生效步骤。
- 携带资产类型/谱系/数量状态、采集交互预留和不可中断边界。
- 会影响玩法资格的来源标签。

以下不得进入玩法键：

- 运行时对象地址、数组位置、字典顺序和临时批次ID。
- 画面代表、动画、镜头、页面和帧率。
- 只用于统计且不改变行为的生产事件身份。
- 可从事件账本与当前份额重算的派生摘要。

### 3.2 生产归因

统计性生产归因使用与状态单元并列的稳定稀疏份额表：

```text
sum(各归因键数量) = 状态单元真实数量
```

合并时同键相加，拆分时按归因稳定顺序和精确前缀数量分配，行列总和必须守恒。若后续内容使生产来源影响伤害、目标或资格，该来源标签必须提升为玩法键，不得继续作为纯统计份额。

### 3.3 唯一规范序列

每次可能改变键的事务后统一执行：

1. 删除数量为0的单元，但保留事件和累计账目。
2. 终结已过期状态并重建状态多重集。
3. 规范化定点数、标签、目标、路径、状态和归因顺序。
4. 以完整玩法键排序。
5. 对键完全相同的相邻单元求和，并合并归因份额。
6. 检查总数量、总耐久、状态层数、携带资产、行为承诺和归因守恒。

规范化必须幂等：`normalize(normalize(S)) = normalize(S)`。序列化前后和不同合法初始分片都必须得到相同规范哈希。

## 4. 状态效果规范化

每个状态定义必须登记：

- 稳定效果ID、幅度量子和作用字段。
- `刷新持续 / 取最强 / 独立实例 / 固定叠层`之一。
- 生效/失效步骤、有限持续或明确来源结束条件。
- 来源保留模式、叠加上限或可证明的有限应用包络。
- 是否具有终止触发；第一版终止触发不能递归形成同类状态。

规范表示：

| 叠加方式 | 权威表示 |
| --- | --- |
| 刷新持续 | 每个效果键一个实例，保存规范幅度、最终失效步骤和来源份额 |
| 取最强 | 保存全部尚未失效候选的聚合多重集；有效项是派生结果，不能删除暂时较弱项 |
| 独立实例 | 按效果、幅度、失效步骤和后续负载键聚合相同实例数量 |
| 固定叠层 | 保存0至声明上限的整数层和统一/逐层失效策略 |

相同实例可以按数量合并，不同失效步骤不能取平均。被动范围在每步由存活来源与位置重算，不复制进永久状态队列；来源失效后派生修正消失。

## 5. 无损拆分与合并

### 5.1 拆分

从数量`N`的状态单元拆出`k`时：

- `0 <= k <= N`，产生数量`k`与`N-k`的两个精确单元。
- 完整玩法键先复制，随后只对事件声明改变的字段生成新键。
- 归因、携带资产、状态实例数量和行为承诺使用稳定前缀分配，分别满足行列守恒。
- 不通过比例浮点或独立四舍五入分配整数余数。

### 5.2 合并

只有完整玩法键一致才能合并。合并只做数量和统计份额相加，不平均耐久、进度、持续、目标、路径或资产。用于性能的容器分片不是玩法键，必须在规范化时消失。

### 5.3 表示等价

给定同一规范状态，任意合法预分片、并行容器和插入顺序在执行相同命令后必须产生相同：

- 规范状态哈希。
- 事件排序与账目。
- 伤亡、变形、状态和死亡输出。
- 保存快照与重放结果。

## 6. 批量命中算法

### 6.1 单一等价攻击包

对一个稳定目标池，设规范排序后的状态单元数量为`n_i`，池总逻辑个体`N=sum(n_i)`，命中槽数`K`：

```text
q = K div N
r = K mod N
prefix_i = sum(n_j), j < i
extra_i = clamp(r - prefix_i, 0, n_i)
```

- 单元中`extra_i`个逻辑个体获得`q+1`次命中。
- 其余`n_i-extra_i`个获得`q`次命中。
- 每个原单元最多因该攻击包形成两个后继单元。
- `N=0`时攻击包失效但保留来源行为事件，不发生除零。

排序使用空间暴露、距离、耐久和完整玩法键；运行批次ID、来源事件顺序和画面对象不参与。玩法键完全相同的单元必须先合并，因此不存在需要按容器顺序裁决的同键目标。

### 6.2 多种同时攻击包

同一步第9阶段：

1. 先按目标池、有效伤害/状态配方、覆盖和作用空间规范合并等价攻击包。
2. 每类攻击包都读取第8阶段同一目标快照并计算商余分配。
3. 收集每类分配在规范逻辑序列上的边界，形成精确区间叠加。
4. 每个区间得到完整命中/状态向量，一次求值为后继状态。
5. 所有区间同时提交，再规范合并相同后继单元。

不能把一个攻击包提交后产生的受伤队列作为同一步另一个攻击包的新目标快照。过量伤害留在已分配逻辑个体上，只有攻击包预先声明的穿透、范围或连锁可以增加覆盖。

### 6.3 状态与变形覆盖

有限覆盖的状态、变形和其他单位效果使用同一规范序列与商余分配。需要每目标独立随机的效果不属于第一版；后续若加入，必须使用可跳跃、可按区间求值的确定性随机流，不能逐个遍历百万单位。

## 7. 来源、死亡和残骸

- 耐久降到0及以下的后继区间形成聚合死亡数量，不创建逐单位死亡对象。
- 死亡输出按模板、节点位置、谱系和参数版本聚合为过程资产批次。
- 统计归因按死亡区间从稳定来源份额表精确扣减。
- 一个死亡事件幂等键覆盖聚合区间及来源步骤，重试、读档和表现回调不能重复形成残骸。
- 后台玩家固定单位死亡仍按内容规则形成零可回收输出，但数量与统计正常终结。

## 8. 复杂度边界

设：

- `C`：规范状态单元数。
- `S`：规范状态实例条目数。
- `A`：本步骤等价攻击/效果包数。
- `V`：统计归因键总数。
- `O`：多包区间叠加形成的后继单元数。
- `N`：真实逻辑单位总数。

目标复杂度：

```text
时间 = O((C + S + A + O) log C + V)
内存 = O(C + S + A + O + V)
```

除固定宽度整数运算外，时间和内存不得与`N`线性相关。`O`可能因内容组合增长，因此发布内容必须通过可达状态和压力夹具验证；不能假设“状态通常很少”。

## 9. 第一版内容包络

第一版内容满足：

- 每个模板至多一个核心行为。
- 战斗无随机命中、暴击和浮动伤害。
- 当前实际持续状态族只有共振亢奋；根网是按位置重算的被动范围，不进入持久状态多重集。
- 候选只修改已有行为、配方、目标、负载或被动范围，不创建第二行为。
- 所有持续、叠层、覆盖、来源储备、行为周期和空间容量具有有限参数。
- 任何新增状态必须由内容校验器证明其幅度种类、失效步骤窗口、叠加和来源数量在发布夹具内有界。

通用内核仍要用合成夹具分别覆盖四种状态叠加模式。合成夹具证明接口，不把未使用状态加入第一版内容。

## 10. 运行时超预算处理

状态单元和状态实例没有会改变语义的运行硬上限。发布预算只用于内容、参数和性能验收。

当未发布完整步骤超过单个表现帧的工作片预算时：

1. 保持上一个稳定边界为唯一可读玩法状态。
2. 把确定性工作游标保存在临时步骤工作区，跨现实帧继续同一步。
3. 不推进部分模拟时间，不发布部分伤亡，不接收可插入本步骤中间的命令。
4. 完整第12阶段校验通过后一次发布。

若工作区达到内存安全看门限、算术溢出或确定性工作上限：

- 丢弃整个未发布工作区。
- 恢复上一个稳定边界并进入诊断暂停。
- 保存可复现输入、状态哈希、内容/参数版本和失败原因。
- 不裁剪队列、不平均状态、不跳过效果，也不把错误状态继续保存为正常存档。

## 11. 基准夹具

### 11.1 常规百万级夹具 `million_normal`

- 总存活单位：2,000,000，敌我各1,000,000。
- 规范状态单元：最多1,024个测试输入单元。
- 规范状态实例条目：最多4,096个。
- 等价攻击/效果包：每步最多128类。
- 统计归因键：最多256个。
- 覆盖接触、纵深、后备、工蜂、固定菌毯、出兵源和同时死亡。

### 11.2 压力夹具 `million_stress`

- 总存活单位：4,000,000，敌我各2,000,000。
- 规范状态单元：最多4,096个测试输入单元。
- 规范状态实例条目：最多16,384个。
- 等价攻击/效果包：每步最多512类。
- 统计归因键：最多1,024个。
- 同时覆盖四种状态叠加、不同耐久、变形、死亡输出、进攻预留释放和原子撤离。

这些数字定义需要支持的测试包络，不是单位、状态或伤害的玩法上限。合法运行状态超过该包络时仍必须精确，只是发布前需要新增更高档证据。

## 12. 候选性能预算

预算在`perf.reference_machine_id`登记的第一版桌面参考机器上测量。参考记录必须包含CPU、内存、操作系统、Godot版本、构建模式、电源策略和后台负载。缺少参考机器时不能产生`release_verified`性能证据。

| 指标 | 候选发布预算 |
| --- | ---: |
| `million_normal`完整步骤P95 | <=16 ms |
| `million_normal`完整步骤P99 | <=25 ms |
| `million_stress`完整步骤P95 | <=50 ms |
| `million_stress`完整步骤P99 | <=100 ms |
| 压力夹具进攻或撤离事务P95 | <=50 ms |
| 压力夹具规范快照编码P95 | <=250 ms |
| 压力夹具权威模拟状态峰值 | <=256 MiB |
| 压力夹具单份未压缩规范快照 | <=64 MiB |

测量要求：

- 发布构建，固定CPU亲和/电源策略，预热1000步。
- `million_normal`测量10000步，`million_stress`测量2000步。
- 每个夹具至少10次独立进程重放，规范状态与事件哈希完全一致。
- 报告中位数、P95、P99、最大值、峰值内存和垃圾回收/分配次数。
- 长时循环后状态单元、内存和快照大小只能随真实状态变化，不得因历史步骤无界增长。

候选预算可以在首次正式基准前根据参考硬件调整，但必须登记理由；看到失败结果后降低门槛不能继续使用同一证据版本。

## 15. 完成条件

- 规范状态单元、状态效果、拆分、合并、商余命中和同时效果存在唯一可执行定义。
- L3不再登记“状态队列上限策略”，只登记无损复杂度与性能预算。
- 第一版内容的可达状态组合有限且通过静态校验。
- 两个百万级夹具满足候选时间与内存预算，并与小规模逐逻辑个体参考模型完全等价。
- 超预算和安全看门限只影响现实等待或诊断状态，不改变玩法结果。
- L6Q-001至L6Q-012达到`release_verified`后，E-03才允许进入发布体验验证。
