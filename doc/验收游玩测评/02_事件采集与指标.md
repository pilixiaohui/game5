# game5 事件采集与指标

## 1. 采集目标

事件用于回答“玩家实际经历了什么”，不能用于替代存档权威状态，也不能改变模拟顺序、随机数、帧率或资源结果。

必须同时保留三类来源：

- `game`：由权威游戏状态在命令提交或事实形成后记录。
- `harness`：由启动器记录构建、进程、帧时间、内存、崩溃和文件操作。
- `observer`：由观察员记录任务提示、帮助、停滞、言语和主动停止。

游戏内当前 `ledger` 只保留最近 80 条领域事实，不能作为一小时终验的唯一原始数据。终验需要独立、追加写入、按场次保存的 JSONL 文件。

## 2. 事件公共信封

每行是一个完整 JSON 对象。字段名和枚举使用稳定英文 ID；中文显示文本只能放在 `payload.display_text`，不得作为分析主键。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `schema_version` | string | 是 | 固定为 `game5.acceptance.event.v1` |
| `session_id` | string | 是 | 一次连续进程场次的唯一 ID |
| `participant_id` | string | 是 | 匿名参与者 ID |
| `sequence` | int | 是 | 从 1 开始、严格递增且不重复 |
| `source` | enum | 是 | `game / harness / observer` |
| `event_name` | string | 是 | 来自本文件注册表 |
| `result` | enum | 是 | `fact / accepted / rejected / failed / cancelled` |
| `reason_code` | string | 条件 | 拒绝、失败或停止时必填；不得只记录自然语言 |
| `wall_time_utc` | string | 是 | ISO 8601 UTC 时间 |
| `run_ms` | int | 是 | 从本场进程启动开始的单调时间，不受系统时钟调整影响 |
| `sim_tick` | int | 游戏事件必填 | 权威模拟 tick |
| `sim_speed` | int | 游戏事件必填 | `1 / 2 / 4` |
| `build_commit` | string | 是 | 完整 Git commit |
| `build_sha256` | string | 是 | 被测可执行包哈希 |
| `seed` | int | 有存档后必填 | 当前轮回种子 |
| `save_generation` | int | 有存档后必填 | 每次成功保存递增，用于跨进程对应 |
| `page_id` | string | 条件 | `title / hive / map / battle / evolution / swarm / ascension / system` |
| `command_id` | string | 命令事件必填 | 一次请求与结果共用的关联 ID |
| `payload` | object | 是 | 事件专属字段；无字段时使用 `{}` |
| `state_digest` | object | 关键事实必填 | 事件提交后的最小状态摘要 |

禁止写入姓名、邮箱、系统用户名、完整本地路径、IP 地址或未授权的设备标识。

### 2.1 JSONL 示例

```json
{"schema_version":"game5.acceptance.event.v1","session_id":"S-20260715-001","participant_id":"P-N-004","sequence":42,"source":"game","event_name":"battle_retreated","result":"fact","reason_code":"","wall_time_utc":"2026-07-15T08:12:31.442Z","run_ms":754221,"sim_tick":183,"sim_speed":2,"build_commit":"f2ef36937a106279fa50262da50c1d0fcaca51b5","build_sha256":"<sha256>","seed":2707,"save_generation":3,"page_id":"battle","command_id":"CMD-0018","payload":{"node_id":"B","returned_biter":14,"returned_spore":2,"fixed_roots":3},"state_digest":{"resources":{"biomass":61.0,"energy":88.0,"genes":0.0},"active_battle":null,"owned_nodes":["H"]}}
```

## 3. `state_digest` 最小内容

以下事件必须携带提交后的摘要：命令结果、房间完成、单位形成、战斗开始/结束、样本阶段变化、候选变化、保存/载入和状态校验失败。

| 分组 | 字段 |
| --- | --- |
| 资源 | `biomass`、`energy`、`genes` |
| 能源 | `supply`、`base_load`、`task_load`、`net`、`capacity` |
| 过程资产 | `field_organic`、`field_carcass`、`secured_carcass`、`field_sample`、`secured_sample`、`sample_tissue` |
| 单位 | `worker`、`biter`、`root_spore`、`root_mat`、`formed`、`lost` |
| 房间 | 各状态数量及发生事件的槽位完整状态 |
| 地图 | `owned_nodes`、`observed_nodes`、发生事件节点的敌军与结构值 |
| 战场 | 节点、双方期初/当前数量、歼灭、损失、经过 tick；无战场为 `null` |
| 进化 | `candidate_group_id`、候选 ID、已同化 mutation ID |
| 进度 | 已完成 `FH-*` ID、相邻区域是否解锁 |

摘要用于快速核对，完整状态仍以保存时归档的权威存档为准。

## 4. 命令事件契约

每个玩家命令产生一对事件：

1. `command_requested`：记录公开命令 ID 和参数，不得包含尚未揭示的信息。
2. `command_resolved`：记录 `accepted / rejected / failed`、稳定 `reason_code` 和提交后摘要。

支持的命令 ID 至少包括：

| `command_name` | 关键参数 |
| --- | --- |
| `new_game` | `seed_source` |
| `set_speed` | `speed` |
| `build_room` | `slot`、`room_kind` |
| `cancel_room` | `slot` |
| `demolish_room` | `slot` |
| `move_room` | `from_slot`、`to_slot` |
| `set_room_paused` | `slot`、`paused` |
| `set_unit_target` | `unit_kind`、`old_target`、`new_target` |
| `attack_node` | `node_id` |
| `retreat` | `node_id` |
| `induce_candidate` | `source` |
| `culture_candidate` | `source_room_slot` |
| `select_candidate` | `group_id`、`candidate_id` |
| `save_game` | `save_generation` |
| `load_game` | `requested_generation` |

拒绝原因必须使用如 `insufficient_biomass`、`room_locked`、`slot_occupied`、`battle_already_active`、`no_combat_units` 等稳定 ID。显示文案变化不能改变分析结果。

## 5. 游戏领域事件注册表

`实现状态` 中的“已有”表示当前 `game_session.gd` 已调用 `_record`，但仍需接入独立 JSONL 信封。

| event_name | 实现状态 | 必需 payload | 主要用途 |
| --- | --- | --- | --- |
| `round_started` | 已有 | `seed` | 新档起点 |
| `room_build_started` | 已有 | `slot, room_kind, biomass_reserved` | 建设顺序、资源投入 |
| `room_build_cancelled` | 已有 | `slot, room_kind, biomass_refunded, elapsed_ticks` | 取消守恒 |
| `room_completed` | 已有 | `slot, room_kind, build_ticks` | 建设耗时 |
| `room_demolished` | 已有 | `slot, room_kind, biomass_refunded` | 布局重构 |
| `room_moved` | 已有 | `from_slot, to_slot, room_kind` | 邻接与试错 |
| `room_policy_changed` | 已有 | `slot, paused` | 产能管理 |
| `room_group_changed` | 必增 | `group_id, room_kind, slots` | 合并房间是否形成 |
| `room_blocked` | 必增 | `slot, block_reason` | 缺料、缺能、堵塞 |
| `room_unblocked` | 必增 | `slot, previous_reason, blocked_ms` | 恢复路径 |
| `energy_threshold_changed` | 必增 | `from_band, to_band, net` | 能源供需理解 |
| `unit_target_changed` | 必增 | `unit_kind, old_target, new_target` | 生产决策 |
| `unit_formed` | 已有 | `unit_kind, amount, room_slot` | 形成节奏 |
| `biomass_delivered` | 已有 | `amount, room_slot, worker_count` | 资源形成链 |
| `field_asset_secured` | 必增 | `asset_kind, amount, worker_count` | 工蜂采集 |
| `carcass_digested` | 已有 | `amount_in, biomass_out, room_slot` | 残骸链 |
| `sample_separated` | 已有 | `amount, room_slot` | 样本第一阶段 |
| `sample_analyzed` | 已有 | `amount, genes_out, room_slot` | 样本第二阶段 |
| `battle_started` | 已有 | `node_id, biter, spore, enemy, structure` | 进攻期初 |
| `battle_sampled` | 必增 | 当前双方数量、损失、经过 tick | 战况曲线 |
| `battle_retreated` | 已有 | 期初、本期死亡、撤回、留置、敌方恢复 | 撤离结算 |
| `army_destroyed` | 已有 | `node_id, formed, lost, enemy_remaining` | 失败恢复 |
| `node_captured` | 已有 | `node_id, survivors, loot, external_source` | 推进漏斗 |
| `enemy_recovery_changed` | 必增 | `node_id, before, after, cause` | 节点持久性 |
| `candidate_group_formed` | 已有 | `group_id, kind, source, option_ids` | 候选来源 |
| `mutation_assimilated` | 已有 | `group_id, selected_id, discarded_ids, genes_cost` | 单选消费 |
| `mutation_effect_applied` | 已有 | `mutation_id, context, before, after` | 构筑是否生效 |
| `adjacent_region_revealed` | 必增 | 相邻区域公开字段 | 出口结果 |
| `ascension_preview_opened` | 必增 | `points, contribution_digest` | 终局预览发现 |

`battle_sampled` 按固定模拟间隔采样，不按渲染帧采样。开始、撤离、全灭和占领必须保留完整事实，不能被采样覆盖。

## 9. 派生指标

所有时间指标使用 `run_ms`；模拟产出使用 `sim_tick`。二者不可混算。

| metric_id | 定义 |
| --- | --- |
| `TIME_FIRST_ROOM` | `room_build_started.run_ms - round_started.run_ms` |
| `TIME_FIRST_WORKER` | 首个 `unit_formed(worker)` 的运行时间 |
| `TIME_FIRST_ARMY` | 首次同时拥有可进攻战斗单位的运行时间 |
| `TIME_FIRST_ATTACK` | 首个已接受 `attack_node` 的运行时间 |
| `TIME_FIRST_CAPTURE` | 首个虫巢外 `node_captured` 的运行时间 |
| `TIME_FIRST_MUTATION` | 首个 `mutation_assimilated` 的运行时间 |
| `TIME_EXIT_CAPTURE` | 出口 `node_captured` 的运行时间 |
| `TIME_ASCENSION_VIEW` | 首个 `ascension_preview_opened` 的运行时间 |
| `NATURAL_DISCOVERY_RATE` | 在对应 `task_prompted` 前已经完成该目标的参与者比例 |
| `INDEPENDENT_TASK_RATE` | 完成任务且最高帮助等级不超过 `H1` 的比例 |
| `STALL_TIME_RATIO` | 所有停滞时长 / 实际游玩时长；有重叠时只计一次 |
| `BLOCKED_ROOM_TIME` | 各房间 `room_blocked` 至 `room_unblocked` 的模拟时长 |
| `ENERGY_ZERO_TIME` | 储能处于 0 且净变化不为正的模拟时长 |
| `MEANINGFUL_ACTION_RATE` | 建房、生产目标、进攻、撤离、候选选择等策略命令数 / 游玩分钟 |
| `BATTLE_SURVIVAL_RATE` | 撤离或占领后存活可机动单位 / 战役期初可机动单位 |
| `RETRY_COST` | 同节点两次进攻间形成单位、生物质消耗和真实等待时间 |
| `ROUTE_SIGNATURE` | 按首次占领顺序形成的节点 ID 序列 |
| `BUILD_SIGNATURE` | 前六个房间建设/拆除事实及首次突变 ID |
| `CONTENT_EXHAUSTION_TIME` | 玩家首次明确无有意义目标时的 `run_ms`；未耗尽为右删失 |
| `VOLUNTARY_45M` | 第 45 分钟仍自愿游玩且未被任务强留 |
| `SAVE_CONTINUITY` | 载入后关键状态摘要与保存成功摘要逐字段一致 |

页面切换、保存和设置不是策略行为，不计入 `MEANINGFUL_ACTION_RATE`。有意识观察战斗可以是有效体验，但不能虚构为命令数量。

## 10. 数据质量门槛

- 每个场次必须具有 `session_started` 和且仅一个终止事件。
- `sequence` 连续递增，`run_ms` 和同一进程内 `sim_tick` 不倒退。
- 每个 `command_requested` 必须有同 `command_id` 的唯一 `command_resolved`。
- 每次成功保存前后必须有摘要，重启后第一个载入摘要必须可对应保存代数。
- 关键转移事件不能仅存在于录像或本地化文本中。
- JSONL 每行可独立解析；坏行不得静默跳过。
- 写入失败必须触发可见 `telemetry_failed`，终验不能在未知数据缺口下继续签发通过。
- 原始日志、录像、存档和观察表生成 SHA-256 并在报告中登记。

关键事件缺失时，受影响指标标记为 `INVALID`，不能按 0 次发生处理。若缺失影响硬门槛或核心漏斗，整场不能计入终验样本，并须登记测试基础设施缺陷。

## 11. 当前实现差距

当前领域事件定义位于 [game_session.gd](../../variants/desktop_v1/scripts/core/game_session.gd)，但终验前仍需封闭：

- 独立、无上限替换的 JSONL 事件出口。
- 公开命令请求/结果及稳定拒绝码。
- 页面访问、速度、单位目标和飞升预览访问。
- 能源阈值、房间阻塞、工蜂保障资产和战场采样。
- 保存成功/失败、写盘耗时和跨进程代数。
- 进程、性能、崩溃、主持人帮助、停滞和内容耗尽。

这些差距不表示玩法一定错误，但在补齐前无法形成可审计的最终游玩验收证据。
