# 《异种起源：无尽洪流》第一版验收游玩测评包

## 1. 目的

本目录用于验收 `variants/desktop_v1/` 桌面第一版。它把技术测试、规则验证、真人操作、主观体验、内容容量、无障碍和缺陷管理放入同一证据链，但不把它们压缩成一个可以互相抵消的总分。

正式结论只允许为：

- `PASS`：全部硬门槛通过，体验门槛达到冻结值。
- `CONDITIONAL PASS`：没有阻断缺陷，但存在已批准、有限且必须复测的偏差。
- `FAIL`：任一硬门槛失败，或证据不足以支持终验。

形成性试玩只能发现问题和校准指标，不能签发 `PASS`。

## 4. 证据优先级

发生冲突时按以下顺序处理，不使用平均分掩盖冲突：

1. 崩溃、卡死、数据损坏、状态不守恒等客观事实。
2. 游戏日志、存档、固定种子回放和事件序列。
3. 录像中可定位的玩家行为、停滞、误操作和求助。
4. PXI 各维度分数与结构化访谈。
5. 测试人员或开发者的解释。

玩家声称“目标很清楚”但长时间无法判断下一步时，应登记为证据冲突并复查，不能仅采用问卷答案。

## 5. 测试角色

| 角色 | 职责 | 限制 |
| --- | --- | --- |
| 验收负责人 | 冻结版本、样本、门槛并签发结论 | 不得在看到终验结果后降低门槛 |
| 构建管理员 | 生成包、哈希、干净用户目录和日志 | 不主持自己临时修改的构建 |
| 主持人 | 朗读脚本、控制帮助等级和时间 | 不解释设计意图，不推荐路线或构筑 |
| 观察员 | 记录事件、停滞、言语和问题 | 不与玩家交谈 |
| 数据分析员 | 校验事件、计算指标、PXI 和区间 | 不删除不利样本；排除必须有理由 |
| 缺陷负责人 | 去重、分级、指派和关闭问题 | 严重度与修复优先级分开管理 |

人员不足时可以兼任，但主持人不应是该场次被测功能的直接实现者。

## 6. 玩家分层

终验至少区分以下人群，结果不得只汇总为一个总体平均值：

- `NOVICE`：未接触过 game5，对放置、自动化或经营策略类游戏经验有限。
- `GENRE`：未接触过本次构建，但熟悉放置、自动化、经营或策略游戏。
- `RETURNING`：完成过一次规定场次，在约定间隔后使用原存档返回。
- `ACCESS`：具有本轮目标无障碍需求的玩家；真实需求测试不能只由开发者模拟替代。

参与开发、阅读过设计文档、看过他人测试或玩过同一构建的人员不能作为 `NOVICE` 或 `GENRE` 终验样本。

形成性轮次可从每个同质分层约 5 至 6 人开始，并进行多轮修订。终验样本量必须根据先导轮方差、预期差异和统计功效登记；未做样本量说明的结果只能算形成性证据。

## 7. 门槛冻结

门槛分为两类：

- 规格门槛：崩溃、存档、规则守恒、最低分辨率等，不因先导数据降低。
- 体验门槛：任务完成率、PXI、内容容量等，可在先导轮校准一次，终验构建发放前冻结。

冻结后若必须改变门槛，应作废当前终验，登记原因并重新抽取样本。不能在看到终验结果后把失败线移动到结果以下。

## 8. 隐私与数据保管

- 参与者使用随机 `participant_id`，事件日志不记录姓名、账号、邮箱或系统用户名。
- 录像、声音、出生年和无障碍信息必须单独征得同意。
- 存档路径、计算机名和本地目录进入报告前需要脱敏。
- 原始数据只读保存；清洗、排除和派生指标在副本上完成并保留审计记录。
- 参与者可以随时停止，提前停止本身是有效结果，不得强迫其玩满一小时。
