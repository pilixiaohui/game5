| 项目 | 内容 |
| --- | --- |
| 文档版本 | 1.0 |
| 文档层级 | L6，测试与发布证据 |

| 当前状态 | 协议为`已确认规则`；E-01至E-06仍为`设计假设`，当前无发布证据 |
| 适用范围 | 第一版首小时体验验证 |
| 直接依赖 | `L1_设计简报与体验目标.md`、全部L3/L4、两份L5、`L6_无表现模型与验收映射.md` |

本协议把“想要的体验”转换为可被支持或反驳的证据。它不能用玩家访谈替代守恒、确定性和性能测试，也不能用自动测试替代玩家是否理解和感受到系统。

---

## 1. 证据层级

每个体验假设必须同时取得三类证据：

| 证据层 | 回答问题 | 发布要求 |
| --- | --- | --- |
| 形式与无表现 | 机制是否守恒、确定、可恢复且不依赖表现 | 对应INV、L3和L4自动验收全部通过 |
| 行为任务 | 玩家是否能独立完成、识别因果并作出选择 | 达到本协议预注册人数与通过线 |
| 主观体验 | 玩家是否实际感到控制、规模、代价、自由和连续性 | 达到对应构念问卷与访谈通过线 |

任何一层失败都不能由另两层平均抵消。形式错误属于发布阻断；玩家任务或体验失败则反驳对应假设，需要修改机制、内容、参数或反馈后重新验证。

## 2. 测试前置门槛

发布体验验证开始前必须冻结并登记：

- Godot可执行构建ID与源码提交标识。
- L2/L3接口基线、L4内容版本、L5协议版本。
- 规则、状态、参数、生成和存档版本。
- 参数表发布候选值与证据状态。
- 首轮种子池、候选随机流池和分配方案。
- 桌面测试设备档、分辨率、画质和代表实体配额。
- 问卷、访谈问题、观察编码表和分析脚本版本。

对应第一版INV、L3、L4内容引用、地图合法性、恢复搜索和表现独立性测试必须至少达到`fixture_verified`。存在已知资源复制、单位失真、永久软锁、地图非法或存档损坏时，不进行发布体验验证。

## 3. 参与者分组

### 3.1 新玩家组

必须同时满足：

- 未阅读本项目设计文档、测试脚本或策略说明。
- 未玩过本项目任何可玩构建。
- 对系统经营、放置、策略或自动战斗游戏至少有基础兴趣，符合L1目标玩家范围。
- 不参与本轮开发、QA或参数调整。

### 3.2 熟悉玩家组

必须满足：

- 在非发布证据构建中完整完成过至少两次第一版机制流程。
- 理解房间、工蜂、进攻、撤离、样本和候选基础操作。
- 未见过本次分配的世界种子和候选随机结果。
- 不参与当前发布参数或关卡内容配置。

熟悉组可以来自早期形成性参与者，但同一人的形成性记录和发布记录不能合并；发布组仍按当前冻结构建重新测试。

## 4. 样本计划

### 4.1 形成性阶段

执行两轮，每轮：

- 5名新的新玩家。
- 3名熟悉玩家。
- 每轮后允许修改机制、参数、界面和协议。

形成性样本只用于发现问题和修订，不计算E-01至E-06发布通过率。第二轮不能用第一轮相同新玩家冒充首次接触。

### 4.2 发布验证阶段

冻结版本后一次性登记：

- 新玩家有效样本`n=12`。
- 熟悉玩家有效样本`n=12`。
- 至少4个已经通过生成校验的首轮种子，每组每个种子最多3人。
- 候选随机流按区组平衡，避免一个候选组覆盖全部参与者。

`12+12`是第一版产品门槛，不是对全部潜在玩家的统计推断。后续公开测试需要更大样本重新估计完成率和体验分布。

### 4.3 有效与无效会话

只有以下外部原因可以在查看结果前判为无效并补招：

- 录制或遥测设备在关键任务开始前完全失败。
- 参与者不符合预先登记的组别条件。
- 主持人意外给出策略答案，污染尚未执行的任务。
- 外部中断导致无法继续，且与产品无关。

游戏崩溃、软锁、存档损坏、性能不可用、界面不可达和参与者主动放弃都是真实失败，不得剔除。所有排除原因、时间和批准人必须在证据登记中保存。

## 5. 主持规则

主持人可以：

- 重述当前任务目标。
- 解释测试设备的非游戏操作。
- 回答“这个文字是什么意思”时读取界面已有定义。
- 在外部设备故障后恢复测试。

主持人不能：

- 推荐房间、位置、单位、比例、节点、候选或撤离时机。
- 指出当前瓶颈、最优路线或隐藏公式。
- 为参与者执行玩法命令。
- 暗示某个选择是测试预期答案。

参与者在没有策略提示下完成记为`independent_pass`；只在允许的事实澄清后完成记为`factual_help_pass`；需要策略提示、主持操作或无法完成记为`fail`。发布行为通过线只计`independent_pass`。

## 6. 任务与数据采集

任务使用L5的T-01至T-07。每个会话按实际可达状态执行，不强制唯一操作顺序。

必须采集：

- 全部命令、拒绝、查询、页面路径、稳定步骤和现实时间。
- 房间、资源、能源、单位、战役、资产、候选与里程碑事件。
- 屏幕与输入录制，或等价可复核证据。
- 主持观察、事实帮助、策略提示和参与者口述。
- 每个任务结果、停滞、恢复、放弃和继续原因。
- E问卷项目原始答案，不只保存平均值。

个人身份与联系方式单独保存，不进入仓库中的玩法证据。

## 7. 主观构念量表

每项使用1至5分：`完全不同意 / 不同意 / 一般 / 同意 / 完全同意`。每个E构念由三项平均，反向题先反向计分。

### 7.1 E-01 活体生产生态

- E01-Q1：我能从房间、工蜂和端口状态判断生产为什么推进或停滞。
- E01-Q2：改变房间或运行状态后，我能看见对应链路发生变化。
- E01-Q3R：除了顶部数字，我看不出虫巢内部发生了什么。

### 7.2 E-02 间接统御

- E02-Q1：即使不能逐个控制单位，我仍能通过生产和构筑影响战况。
- E02-Q2：我能指出某个单位或突变如何改变了实际结果。
- E02-Q3R：战斗结果与我的选择没有可辨认关系。

### 7.3 E-03 吞噬与规模

- E03-Q1：虫群从少量个体增长为宏观群体的变化是连续可感知的。
- E03-Q2：我能区分真实数量和画面中的代表单位。
- E03-Q3R：数量变大后，我无法判断前线、后备和损耗发生了什么。

### 7.4 E-04 真实代价与恢复

- E04-Q1：继续战斗或撤离会带来我能理解的真实后果。
- E04-Q2：遭受重大损失后，我仍能找到继续恢复的可能。
- E04-Q3R：失败要么没有代价，要么只能放弃整个存档。

### 7.5 E-05 构筑自由与适应

- E05-Q1：我看到不止一种可信的生产、路线或突变选择。
- E05-Q2：我的选择改变了损耗、速度、空间或收益关系。
- E05-Q3R：游戏要求我照着唯一顺序和固定构成才能继续。

### 7.6 E-06 连续首小时

- E06-Q1：新的问题和选择在游玩中连续出现，没有依赖固定等待。
- E06-Q2：理解系统后，我可以通过并行和倍率压缩重复过程。
- E06-Q3R：即使已经理解流程，我仍被迫等待同样长的时间。

单项和构念分数都必须保留。缺答不能用组均值补齐；该构念按失败处理，除非整场会话因预先允许的外部原因无效。

## 8. 共同发布门槛

所有E假设共同要求：

- 新玩家关键行为任务至少`10/12 independent_pass`。
- 熟悉玩家同机制覆盖至少`11/12 independent_pass`。
- 对应构念在主要目标组中的中位数至少`4.0/5`，且至少`9/12`参与者构念均值不低于`3.5/5`。
- 资源/单位/战役账目不一致、永久软锁、数据丢失、表现改变结算和隐藏组别补贴均为`0`次。
- 主持人策略提示不计通过。

某个E可以有更严格的专用门槛。任何共同门槛失败即判该E为`release_failed`。

## 9. E-01 活体生产生态

### 9.1 前置证据

- L4R-001至L4R-012及HRC、REE相关第一版验收通过。
- 同一房间链在动画关闭、页面切换和离线等价下账目一致。

### 9.2 行为门槛

- 新玩家T-01与T-04均至少`10/12 independent_pass`。
- 至少`10/12`能在没有策略提示时指出一个真实瓶颈的对象、原因与受影响关系。
- 上述参与者自行执行一个合法操作后，权威数据中对应输入、功率、容量、路径或任务事实必须按规则改变；操作不要求是最优答案。
- 至少`10/12`在复盘时引用房间、工蜂、端口、缓冲或功率事实，而不只说“顶部数字变了”。

### 9.3 反驳

任何一种第一版核心资源需要手动领取、关键停滞只能靠主持人解释、或房间表现与账目不一致，均直接反驳E-01。

## 10. E-02 间接统御

### 10.1 前置证据

- 至少3类单变量对照：生产绑定、单位构成和突变作用。
- 每类使用至少10个合法快照/种子对；A/B只改变一个登记事实，重放结果各自完全确定。
- 至少一个声明指标在A/B间稳定变化，且变化可以追溯到公开接口，不使用隐藏协同倍率。

### 10.2 行为门槛

- 新玩家T-02与T-05均至少`10/12 independent_pass`。
- 至少`10/12`能正确说出工蜂、噬咬体和根脉孢体各自的唯一行为。
- 至少`10/12`能把一次后续推进、损耗或回流变化归因到自己选择的房间、构成或突变，并指出一个可见证据。
- 测试中出现逐单位下令诉求不直接判失败；若超过`2/12`参与者认为没有个体命令就无法影响结果，则E-02失败。

## 11. E-03 吞噬与规模

### 11.1 前置证据

- 同一百万级快照至少在`headless / 最低 / 标准 / 较高`四种代表配额下得到完全相同玩法事件与账目。
- 批次分片、帧率和画质变化不改变目标、命中、死亡、采集或保存。
- 百万级性能必须达到第6项问题最终确定的性能预算后，才能开始E-03发布测试。

### 11.2 行为门槛

- 新玩家T-02和T-07相关数量理解至少`10/12 independent_pass`。
- 至少`10/12`能找到真实精确数量，并明确说明画面代表不是一比一实体。
- 至少`10/12`能正确区分虫群大军、当前节点内移动单位和节点固定留置菌毯，不重复相加。
- 至少`9/12`能根据画面和数据判断一次数量级增长、前线推进方向与损耗趋势。

## 12. E-04 真实代价与恢复

### 12.1 前置证据

- INV-024恢复搜索及无工蜂、零储能、全灭、样本损失、满缓冲和满候选槽组合全部通过。
- 撤离故障注入、战役公式和损失不返还测试全部通过。

### 12.2 行为门槛

- 新玩家T-03至少`10/12 independent_pass`。
- 至少`10/12`在继续或撤离前主动查看一个战线、损耗、敌方来源、残骸或资产事实，并能说明决定依据。
- 使用冻结的极端恢复夹具执行补充任务时，至少`10/12`能在无策略提示下使存档回到一个可继续生产或再次进攻的状态。
- 任何永久软锁、撤离免费返还战损、失败自动回档或恢复依赖未保证稀有候选均直接失败。

## 13. E-05 构筑自由与适应

### 13.1 前置证据

- 至少10000个首轮种子通过地图硬约束和基础完成搜索。
- 每个种子至少两条内部节点不相交的合法出口路径，且基础完成不依赖裂壳体或特定候选。
- 六项候选分别通过作用接口、代价和基础恢复兼容测试。

### 13.2 行为门槛

- 新玩家T-05与T-06均至少`10/12 independent_pass`。
- 24个发布会话中至少出现3种不同的C后支路顺序签名；任一签名不得超过`18/24`。
- 至少两种相同首次候选组各展示给不少于6名参与者；每组都必须出现至少两种不同选择，不能全员选择同一成员。
- 至少`10/12`新玩家能说明自己选择中的一项收益与一项压力。

路线与候选分布只用于发现支配答案，不要求人为平均。未达到多样性门槛时，先检查内容、参数与信息是否形成事实上的唯一解。

## 14. E-06 连续首小时

### 14.1 行为门槛

- 新玩家中至少`10/12`在一次连续会话内触达FH-001至FH-014和FH-UX-001至004。
- 熟悉玩家中至少`11/12`触达完全相同的玩法事件集合，不允许跳过资源、战损、样本或候选链。
- 熟悉玩家完成所需现实时间中位数必须不高于新玩家中位数的`80%`。
- 两组同种子同命令前缀的资源、敌军和候选规则配置完全相同，隐藏产出、伤害、等待或候选补贴为零。
- 固定等待导致熟悉玩家无法通过并行、倍率或快速决策缩短的关键阶段为零。

测试会话可以因观察、访谈和可访问性需要超过一小时；本门槛不把固定分钟写成玩法解锁。首小时发布节奏区间仍由后续参数证据决定。

## 15. 版本变更与重测

发布证据只对完整版本元组有效：

```text
build + state + rules + parameters + content + generation + UX + protocol
```

- 改变机制、内容、参数、关卡、候选、页面路径或反馈后，必须标记受影响E证据`invalidated`。
- 只修正不参与体验的拼写或诊断日志时，可以由两名审查者确认无需重测并记录理由。
- 失败后修改版本必须使用新的参与者重新执行受影响发布样本；不同版本结果不能合并凑过阈值。
- 形成性结果永远不能晋升为发布结果。

## 16. 证据登记

每次发布测试生成：

```text
evidence/ux/<protocol>/<build_id>/
  artifact_index.json
  sessions/<session_id>.json
  telemetry/<session_id>.jsonl
  observations/<session_id>.md
  questionnaires/<session_id>.json
  aggregate/experience_report.json
  aggregate/analysis_log.md
```

`artifact_index.json`保存文件哈希、参与者匿名ID、组别、种子、随机流、有效性和隐私状态。原始记录不可在汇总后删除；公开仓库只保存脱敏索引和摘要，受限录屏按同一证据ID存储。

机器登记：

- `data/validation/l6_experience_manifest.csv.txt`：E-01至E-06协议、样本、任务与门槛。
- `data/validation/l6_experience_task_manifest.csv.txt`：T-01至T-07任务与事件覆盖。
- `data/validation/l6_experience_evidence_registry.csv.txt`：当前证据版本与状态，初始为空证据。

## 17. 判定与报告

每个E只能处于：

| 状态 | 含义 |
| --- | --- |
| `planned_no_evidence` | 协议已登记，尚无有效发布样本 |
| `formative_only` | 有形成性结果，不可发布 |
| `release_collecting` | 冻结版本正在收集，未达到预注册样本 |
| `release_passed` | 全部前置、行为、主观和零容忍门槛通过 |
| `release_failed` | 任一预注册门槛失败 |
| `invalidated` | 原通过证据被相关版本变化失效 |

报告必须逐项列出分子/分母、缺失数据、排除、帮助、原始构念分布、失败场景和版本。不得只报告平均完成时间或总体满意度。

## 18. 第一版体验发布条件

- E-01至E-06全部为同一完整版本元组下的`release_passed`。
- 24个发布会话达到预注册有效样本，不含形成性样本。
- 所有形式前置保持通过，且无零容忍事件。
- E证据登记、原始索引、分析结果和审查签名完整。
- 任何未通过假设都保留为明确反驳结果，不通过改写L1目标或事后降低阈值宣布成功。
